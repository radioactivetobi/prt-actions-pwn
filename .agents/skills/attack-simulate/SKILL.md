# Attack Simulate Skill

Simulate PRT Scan / TeamPCP CI/CD supply chain attacks against GitHub Actions workflows for authorized security research.

## Procedure

### Step 1: Target Analysis

Determine whether the target is a local file, a remote repository, or a demo example:

**Local mode** — target is a file path:
```
Read the target workflow file
Identify trigger type, checkout configuration, permissions model, secrets, action versions, and untrusted input paths
```

**Live target mode** — target matches `owner/repo` format:
```
1. gh api repos/{owner}/{repo} — Target fingerprinting (visibility, language, default branch)
2. gh api repos/{owner}/{repo}/actions/workflows --jq ".workflows[] | .path" — Enumerate all workflows
3. For each workflow, fetch YAML content:
   gh api repos/{owner}/{repo}/contents/{path} --jq ".content" | python -m base64 -d
4. gh api repos/{owner}/{repo}/actions/secrets --jq ".secrets[] | .name" — Identify high-value secrets
5. gh api repos/{owner}/{repo}/branches/{branch}/protection — Check if branch protections block DISPATCH phase
6. gh api repos/{owner}/{repo}/collaborators --jq ".[].login" — Identify accounts for LABEL_BYPASS
7. gh api repos/{owner}/{repo}/environments — Check deployment gates
8. gh api repos/{owner}/{repo}/pulls?state=open --jq ".[] | {number, title, author: .user.login, author_association}" — Find active PRs for injection vectors
```

See `references/gh-recon-commands.md` for the full command reference.

**Demo mode** — target is `demo`:
```
Use live target analysis against the lab repo dextersec-playground/pwn-ci-prt-lab:
1. Fetch workflows via gh api repos/dextersec-playground/pwn-ci-prt-lab/actions/workflows
2. Analyze: vulnerable-prt-target.yml, vulnerable-persist-creds.yml, vulnerable-script-injection.yml
Identify trigger type, checkout configuration, permissions model, secrets, and attack surface
```

### Step 2: Vulnerability Mapping

Map the target to PRT Scan vulnerability patterns (see references/teamPCP-ttps.md):

| Pattern ID | Exploitable Condition | PRT Scan Phase |
|------------|----------------------|----------------|
| PRT-001 | `pull_request_target` + checkout PR head | EXFIL |
| PRT-002 | `persist-credentials: true` (default) | EXFIL |
| PRT-003 | Untrusted input in `run:` blocks | DISPATCH |
| PRT-004 | Missing/broad `permissions:` | RECON, DISPATCH |
| PRT-005 | Unpinned action versions | DELAYED |
| PRT-006 | No first-time contributor approval | LABEL_BYPASS |
| PRT-007 | Exploitable path-based triggers | DISPATCH |
| PRT-008 | Untrusted dispatch inputs | DISPATCH |
| PRT-009 | Secrets in untrusted context | EXFIL, RECON |
| PRT-010 | Forked action references | DELAYED |

Determine which PRT Scan phases are achievable given the target's configuration.

For live targets, use the `gh api` commands from Step 1 to gather:
- Whether `pull_request_target` workflows exist (EXFIL feasibility)
- Whether secrets are accessible (RECON/EXFIL value)
- Whether branch protections exist (DISPATCH feasibility)
- Whether interaction limits exist (LABEL_BYPASS feasibility)
- The repo language (DELAYED injection vector selection)

### Step 3: Injection Vector Selection

Based on the target's language and build system, select the appropriate injection vector:

| Language | Injection File | Mechanism |
|----------|---------------|-----------|
| Python | `conftest.py` | Auto-loaded by pytest before test collection |
| Node.js | `package.json` (scripts field) | Executed by `npm install` / `npm run` |
| Rust | `build.rs` | Compiled and executed before the main crate |
| Go | `_test.go` files | Executed by `go test` |
| Composite | `action.yml` | Referenced by `uses:` in workflows |

For live targets, the repo language from `gh api repos/{owner}/{repo} --jq ".language"` determines the injection vector.

### Step 4: Payload Generation

Generate a 5-phase PRT Scan payload adapted to the target. Each phase must be clearly commented:

```
# ⚠️ SECURITY RESEARCH PAYLOAD - DO NOT DEPLOY ⚠️
# PRT-SCAN-RESEARCH-NONCE-[unique-id]
# Generated by attack-simulate skill for authorized security research only

## Phase 1: EXFIL
# Steal GITHUB_TOKEN via git config
# Detection: Check .git/config for unexpected URLs containing token

## Phase 2: RECON
# Enumerate org repos, branch protections, secrets
# Detection: API calls from GITHUB_TOKEN outside expected workflow steps

## Phase 3: DISPATCH
# Trigger downstream workflows to escalate privileges
# Detection: Unexpected workflow_dispatch/repository_dispatch events

## Phase 4: LABEL_BYPASS
# Auto-apply labels to bypass approval gates
# Detection: Label events from automated accounts without human review

## Phase 5: DELAYED
# Plant backdoor in language-specific config for later execution
# Detection: Unexpected file modifications in conftest.py, package.json, build.rs
```

Use the payload generator script for structured generation:
```bash
python .agents/skills/attack-simulate/scripts/prt-scan-payload.py \
  --target-lang [python|node|rust|go] \
  --phases all \
  --nonce [unique-id]
```

### Step 5: Detection Guidance

For every generated payload, provide detection rules:

```markdown
## Detection Rules for [payload-name]

### Log Indicators
- [Specific log patterns to search for]
- [Unexpected API calls from workflow runs]
- [Branch naming patterns matching PRT Scan IOCs]

### GitHub Alert Rules
- [Recommended alert configurations]
- [Workflow run anomaly detection thresholds]

### IOC Patterns
- Branch names: `prt-*` prefix
- Log markers: `PRT-SCAN-RESEARCH-NONCE-*`
- Network: Calls to 169.254.169.254 from `run:` steps
- File system: Unexpected conftest.py, package.json scripts, build.rs modifications
```

For live targets, include GitHub audit log search queries:

```bash
# Search for unexpected dispatch events
gh api repos/{owner}/{repo}/actions/runs --jq ".workflow_runs[] | select(.event == \"workflow_dispatch\") | {id: .id, actor: .actor.login, created_at: .created_at}"

# Search for label events
gh api repos/{owner}/{repo}/issues/{pr_number}/events --jq ".[] | select(.event == \"labeled\") | {actor: .actor.login, label: .label.name}"
```

## Important Notes

- **NEVER** generate payloads for production repositories — research and lab environments only
- **ALWAYS** mark all generated artifacts with `PRT-SCAN-RESEARCH-NONCE-<id>` comments
- **ALWAYS** confirm with the user before writing any payload files to disk
- **ALWAYS** provide detection guidance alongside any exploitation material
- **NEVER** use real credentials or tokens — all payloads must use placeholder values
- When targeting the demo vulnerable workflows, use live target analysis against `dextersec-playground/pwn-ci-prt-lab`. The demo workflows are: `vulnerable-prt-target.yml`, `vulnerable-persist-creds.yml`, `vulnerable-script-injection.yml`
- Cross-reference the hardening guide (workflow-audit skill) to show how each vulnerability would be remediated
- For live targets, ensure `gh` CLI is installed and authenticated (`gh auth status`)
- Live simulations are analysis-only — do not execute attacks against remote repositories
